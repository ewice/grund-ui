# Tabs — Implementation Spec

Implementation specification for the tabs component and its shared dependency,
`OrderedRegistry`.

---

## Scope

- 1 shared utility: `OrderedRegistry` (extracted from accordion registry, replaces it)
- 1 component-local controller: `TabsController`
- 1 component-local registry: `TabsRegistry` (extends `OrderedRegistry`)
- 1 context interface: root context
- 5 custom elements: `grund-tabs`, `grund-tabs-list`, `grund-tab`, `grund-tabs-panel`, `grund-tabs-indicator`
- No animation — consumers style transitions via `data-activation-direction`, `data-selected`, and `::part()`

### Out of scope

- Vertical overflow scrolling of the tab list
- Drag-to-reorder
- Lazy panel content loading (consumers own this; `keepMounted` is the lever)

---

## 0. OrderedRegistry (shared utility — prerequisite)

New file: `src/utils/ordered-registry.ts`.

Tabs is the second component using the registry pattern. Per the "extract on second use"
rule, extract the shared DOM-ordered list logic before implementing tabs. The accordion
registry is refactored to extend this base. No accordion behaviour changes.

### API

```ts
interface OrderedRecord {
  element: HTMLElement;
}

class OrderedRegistry<TRecord extends OrderedRecord> {
  get entries(): readonly TRecord[];

  insert(record: TRecord): void;
  remove(predicate: (record: TRecord) => boolean): void;
  find(predicate: (record: TRecord) => boolean): TRecord | undefined;
  indexOf(predicate: (record: TRecord) => boolean): number;
}
```

### Behaviour

- `insert` maintains DOM order via `element.compareDocumentPosition(other.element)`.
  Appends to end when no following element exists.
- `remove` filters in place — no DOM queries.
- `find` and `indexOf` take a predicate so callers control the key (element ref for
  accordion, value string for tabs).
- No Lit dependency, no framework dependency.

### Accordion migration

`AccordionRegistry` is refactored to extend `OrderedRegistry<AccordionItemRecord>`.
Public API stays identical. One alias is needed: `AccordionRegistry` exposes `get items()`
which maps to the base class `entries` getter — the name `items` is part of the
accordion's public API and must not change. This is the only change to the accordion
in this scope.

## 1. TabsController

`src/components/tabs/controller/tabs.controller.ts`.
Pure state and action resolution — no DOM access, no Lit dependency.

### API

```ts
interface TabsHostSnapshot {
  value: string | null | undefined;
  defaultValue: string | null;
  disabled: boolean;
}

class TabsController {
  activeValue: string | null;

  syncFromHost(snapshot: TabsHostSnapshot): void;
  requestActivate(value: string, isDisabled: boolean): string | null;
}
```

### Controlled vs Uncontrolled

- **Controlled:** `snapshot.value !== undefined`. `requestActivate` returns the proposed
  new value but does not update `activeValue`. The root fires `grund-value-change`;
  the consumer must update `value`.
- **Uncontrolled:** `defaultValue` seeds `activeValue` once on first `syncFromHost`.
  After that, `requestActivate` updates `activeValue` directly and returns the new value.

### requestActivate

Returns `null` (no change) if:
- `isDisabled` is true
- `value === activeValue` (already active)
- Root `disabled` is true

Otherwise returns the new value string.

### Auto-selection

The controller does not implement auto-selection. The root element handles it in
`firstUpdated` by calling `requestActivate` with the first non-disabled tab's value
from the registry. This keeps DOM knowledge out of the controller.

## 2. TabsRegistry

`src/components/tabs/registry/tabs.registry.ts`.
Extends `OrderedRegistry`. Maps value strings to tab and panel element references.

### API

```ts
interface TabsRecord extends OrderedRecord {
  value: string;
  // element (inherited from OrderedRecord) IS the <grund-tab> host — used for DOM
  // ordering by OrderedRegistry.insert. No separate `tab` field.
  panel: HTMLElement | null;
}

class TabsRegistry extends OrderedRegistry<TabsRecord> {
  registerTab(tab: HTMLElement, value: string): void;
  unregisterTab(value: string): void;

  attachPanel(value: string, panel: HTMLElement): void;
  detachPanel(value: string): void;

  getByValue(value: string): TabsRecord | undefined;
  indexOfValue(value: string): number;
  firstNonDisabled(): TabsRecord | undefined;
}
```

### Key decisions

- Keyed by `value` string — `getByValue` is the primary lookup
- `element` (from `OrderedRecord`) is the `<grund-tab>` host — used for DOM ordering by
  `OrderedRegistry.insert`. There is no separate `tab` field; `element` serves both roles.
- `firstNonDisabled` checks `element.dataset.disabled` to skip disabled tabs
- Both `registerTab` and `attachPanel` use upsert semantics — they create the record if
  absent or update the existing one. This means either element can register first without
  error, regardless of insertion order

## 3. Context

Single context layer. `src/components/tabs/context/tabs.context.ts`.

```ts
interface TabsRootContext {
  activeValue: string | null;
  activationDirection: 'start' | 'end' | 'none';
  orientation: 'horizontal' | 'vertical';
  disabled: boolean;

  registerTab: (tab: HTMLElement) => void;    // caller passes `this`; root reads tab.value before forwarding to registry
  unregisterTab: (value: string) => void;
  registerPanel: (panel: HTMLElement) => void; // caller passes `this`; root reads panel.value before forwarding
  unregisterPanel: (value: string) => void;
  activateTab: (value: string) => void;
  getRegistry: () => TabsRegistry;
}
```

### Design rules

- Single context key symbol: `tabsRootContext`
- `activateTab` is the single action path — keyboard (list) and click (tab) both call it
- `getRegistry` exposes the registry to the indicator element for geometry lookup
- Context is recreated in `willUpdate` whenever `activeValue`, `activationDirection`,
  `orientation`, or `disabled` changes — stable object reference signals "no change"
  to consumers

## 4. Elements

### `<grund-tabs>` — Root

- `@provide` root context
- Owns `TabsController` and `TabsRegistry`
- `willUpdate`: calls `syncFromHost`, recomputes `activationDirection` (index delta
  between previous and new `activeValue`), recreates context if any field changed
- `firstUpdated`: if `activeValue` is `null`, calls `activateTab` with
  `registry.firstNonDisabled()?.value` — activates first non-disabled tab
- `activateTab(value)`: calls `controller.requestActivate`, if non-null computes
  direction from registry index delta, updates `activeValue` + `activationDirection`,
  dispatches `grund-value-change` (controlled fires event only; uncontrolled also
  updates internal state)
- `registerTab` / `unregisterTab` / `registerPanel` / `unregisterPanel`: delegate to
  registry; after each `registerTab`, if `activeValue` is `null`, runs auto-selection
  (covers programmatic tab insertion after first render)
- Render: `<slot>`
- `:host { display: block }`

**Properties:**

| Property | Type | Default | Attribute |
|---|---|---|---|
| `value` | `string \| null \| undefined` | `undefined` | — |
| `defaultValue` | `string \| null` | `null` | `default-value` |
| `orientation` | `'horizontal' \| 'vertical'` | `'horizontal'` | `orientation` |
| `disabled` | `boolean` | `false` | `disabled` |

**Events:**

| Event | Detail | When |
|---|---|---|
| `grund-value-change` | `{ value: string \| null, previousValue: string \| null }` | Active tab changed |

**Data attributes:** `data-orientation`, `data-activation-direction`

---

### `<grund-tabs-list>` — List

- `@consume` root context
- `RovingFocusController` attached — `getItems()` returns all non-disabled `<grund-tab>`
  elements that are slotted children, queried via `this.querySelectorAll('grund-tab:not([data-disabled])')`
- `willUpdate`: sets `aria-orientation`, `data-orientation`, `data-disabled`,
  `data-activation-direction`
- Keyboard handler (`keydown`): `RovingFocusController` owns `ArrowLeft`, `ArrowRight`,
  `ArrowUp`, `ArrowDown`, `Home`, `End` — it handles the `keydown` event and moves
  focus directly. The list's own `keydown` handler runs alongside it (both listen on the
  same element) but only acts on activation:
  - If `activateOnFocus = true`: on any arrow/Home/End key, read the newly focused
    element's `value` and call `ctx.activateTab`. Call `event.preventDefault()` to
    suppress scroll. The list's `keydown` listener is registered after
    `RovingFocusController` adds its own listener in `hostConnected`, so the controller
    always moves focus first — `document.activeElement` reflects the new tab when the
    list's handler reads it.
  - If `activateOnFocus = false`: on Enter/Space, call `ctx.activateTab` with the
    currently focused tab's `value`. Call `event.preventDefault()`.
- Render: `<slot>`
- `:host { display: block }`
- Dev warning if used outside `<grund-tabs>`

**Properties:**

| Property | Type | Default | Attribute |
|---|---|---|---|
| `activateOnFocus` | `boolean` | `true` | `activate-on-focus` |
| `loopFocus` | `boolean` | `true` | `loop-focus` |

**CSS parts:** `list`
**ARIA:** `role="tablist"`, `aria-orientation`
**Data attributes:** `data-orientation`, `data-disabled`, `data-activation-direction`

---

### `<grund-tab>` — Tab

- `@consume` root context
- Renders `<button part="tab" role="tab"><slot></slot></button>`
- `connectedCallback`: calls `ctx.registerTab(this)`
- `disconnectedCallback`: calls `ctx.unregisterTab(this.value)`
- `willUpdate`: sets `aria-selected`, `aria-controls` (panel ID derived from value),
  `aria-disabled`, `data-selected`, `data-disabled`, `data-orientation`, `data-index`,
  `data-activation-direction`
- Click handler: calls `ctx.activateTab(this.value)`
- Panel ID derivation: `grund-tabs-panel-${value}` — must match what `<grund-tabs-panel>`
  sets as its ID
- `:host { display: block }`
- Dev warning if used outside `<grund-tabs-list>`

**Properties:**

| Property | Type | Default | Attribute |
|---|---|---|---|
| `value` | `string` | required | `value` |
| `disabled` | `boolean` | `false` | `disabled` |

**CSS parts:** `tab`
**ARIA:** `role="tab"`, `aria-selected`, `aria-controls`, `aria-disabled`
**Data attributes:** `data-selected`, `data-disabled`, `data-orientation`, `data-index`, `data-activation-direction`

---

### `<grund-tabs-panel>` — Panel

- `@consume` root context
- `connectedCallback`: calls `ctx.registerPanel(this)`; sets its own `id` to
  `grund-tabs-panel-${value}` (deterministic, derived from `value`)
- `disconnectedCallback`: calls `ctx.unregisterPanel(this.value)`
- `willUpdate`: sets `data-selected`, `data-orientation`, `data-activation-direction`
- Visibility is controlled entirely through the shadow DOM render — the panel element
  itself **always stays in the light DOM** (never `remove()`d). The high-level design
  doc's phrase "removed from DOM" refers to the shadow content, not the host element:
  - **Active:** renders `<div part="panel" role="tabpanel" aria-labelledby="${tabId}" tabindex="0"><slot></slot></div>`
  - **Inactive + `keepMounted = true`:** renders `<div part="panel" hidden><slot></slot></div>` — content stays slotted and in the accessibility tree but is hidden
  - **Inactive + `keepMounted = false`:** renders nothing (`html\`\``) — the slot is not present so slotted content is unassigned and not displayed; `role`, `aria-labelledby`, and `tabindex` are absent
- `:host { display: block }`
- Dev warning if used outside `<grund-tabs>`

**Properties:**

| Property | Type | Default | Attribute |
|---|---|---|---|
| `value` | `string` | required | `value` |
| `keepMounted` | `boolean` | `false` | `keep-mounted` |

**CSS parts:** `panel`
**ARIA:** `role="tabpanel"`, `aria-labelledby`, `tabindex="0"` (active only)
**Data attributes:** `data-selected`, `data-orientation`, `data-activation-direction`

---

### `<grund-tabs-indicator>` — Indicator

- `@consume` root context
- No public properties
- Must be placed inside `<grund-tabs-list>` — dev warning if not
- `connectedCallback`: sets up `ResizeObserver` on parent list element; sets up
  `ResizeObserver` on the current active tab element (looked up via
  `ctx.getRegistry().getByValue(ctx.activeValue)?.element`). Both observers call `measure()`.
- On context update (activeValue changes): swaps the tab-level observer to the new
  active tab element; calls `measure()`
- `disconnectedCallback`: disconnects both observers
- `measure()`: if no active tab, no-op. Calls `getBoundingClientRect()` on the active
  tab element and the parent list element. Computes the six offsets. Sets as inline CSS
  custom properties on `this` (the host element):
  - `--grund-tabs-indicator-left`, `--grund-tabs-indicator-right`
  - `--grund-tabs-indicator-top`, `--grund-tabs-indicator-bottom`
  - `--grund-tabs-indicator-width`, `--grund-tabs-indicator-height`
- `willUpdate`: sets `data-orientation`, `data-activation-direction`
- Render: `<span part="indicator"></span>`
- `:host { display: block; position: relative }` — consumers position the inner span

**CSS parts:** `indicator`
**Data attributes:** `data-orientation`, `data-activation-direction`

## 5. Dev Warnings

All guarded with `if (import.meta.env.DEV)`.

| Element | Condition | Message |
|---|---|---|
| `<grund-tabs-list>` | No ancestor `<grund-tabs>` | `[grund-tabs-list] Must be a child of <grund-tabs>.` |
| `<grund-tab>` | No ancestor `<grund-tabs-list>` | `[grund-tab] Must be a child of <grund-tabs-list>.` |
| `<grund-tab>` | Duplicate `value` in registry | `[grund-tab] Duplicate value "${value}". Each tab must have a unique value.` |
| `<grund-tab>` | No matching panel in registry after firstUpdated | `[grund-tab] No <grund-tabs-panel value="${value}"> found.` |
| `<grund-tabs-panel>` | No ancestor `<grund-tabs>` | `[grund-tabs-panel] Must be a child of <grund-tabs>.` |
| `<grund-tabs-indicator>` | No ancestor `<grund-tabs-list>` | `[grund-tabs-indicator] Must be a child of <grund-tabs-list>.` |

## 6. Vocabulary Additions

Register in `docs/vocabulary.md` before merging:

| Attribute | Values | Applies to | Semantics |
|---|---|---|---|
| `data-selected` | boolean presence | `<grund-tab>`, `<grund-tabs-panel>` | Active in a selection group. Use for selection-based components (tabs, radio group, listbox, combobox). Distinct from `data-open` (disclosure/expand patterns). |
| `data-activation-direction` | `'start'` / `'end'` / `'none'` | All tabs elements | Direction of the last activation relative to list order. Logical values — combine with `data-orientation` to derive physical direction. |
