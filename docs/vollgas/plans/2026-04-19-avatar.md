# Avatar Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use vollgas:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship a headless, accessible Avatar primitive (`<grund-avatar>` / `-image` / `-fallback`) matching Base UI's compound API, with web-component-native idioms.

**Architecture:** Root element owns a tiny `AvatarEngine` (pure status reducer), provides `AvatarContext` via `@provide`. Image element reports status from its internal `<img>`'s `load`/`error` events. Fallback element gates its own visibility by status + configurable delay. No random IDs, no ARIA added by components, no color-hashing. Only structural CSS (`:host { display: ... }`). Testable engine is independently instantiable.

**Tech Stack:** Lit 3, `@lit/context`, Vitest + Playwright browser mode (`@open-wc/testing`), Storybook.

**Spec:** [docs/vollgas/specs/2026-04-19-avatar-design.md](../specs/2026-04-19-avatar-design.md)

- Compound API — a root element with specialised sub-part elements.

- [ ] **Step 3: Implement engine**

`src/components/avatar/avatar.engine.ts` — responsible for holding `AvatarStatus` and notifying subscribers on actual transitions. Pure class, no DOM, no Lit import.

Contract:
- `export class AvatarEngine` marked `@internal`.
- `public get status(): AvatarStatus` — current status; starts `'idle'`.
- `public setStatus(next: AvatarStatus): void` — if `next === this.status`, return without notifying. Otherwise update and invoke every subscribed listener with the new status.
- `public onChange(listener: (status: AvatarStatus) => void): () => void` — register a listener and return an unsubscribe function. Implementation: maintain a `Set<Listener>`; unsubscribe removes from the set.
- Invariants: `status` is always one of the four values in `AvatarStatus`; notification count equals number of *actual* transitions.

- [ ] **Step 3: Implement root element**

`src/components/avatar/avatar.ts` — responsible for owning `AvatarEngine`, providing `avatarContext`, reflecting `data-status`, dispatching `grund-status-change`, and rendering a shadow `<div part="root"><slot></slot></div>`.

Contract:
- `export class GrundAvatar extends LitElement`.
- JSDoc header with `@element grund-avatar`, `@slot - Default slot for <grund-avatar-image> and <grund-avatar-fallback>`, `@fires {CustomEvent<AvatarStatusChangeDetail>} grund-status-change - When the image loading status transitions`, `@csspart root - Host container`.
- `public static override readonly styles = css\`:host { display: inline-block; /* inline-block: avatar wraps inline content but itself is a sized box */ }\``.
- No public reactive properties (spec §3).
- `@provide({ context: avatarContext }) @state() protected avatarCtx!: AvatarContext` — reassigned in `willUpdate` when engine status changes (Rule 15).
- Private `engine = new AvatarEngine()` instantiated at field level (no DOM).
- In constructor, subscribe to `engine.onChange` so the element calls `this.requestUpdate()` on transitions. Store the unsubscribe function; call it in `disconnectedCallback`.
- `setStatus` exposed on context as a stable arrow-function field: `private readonly _setStatus = (next: AvatarStatus) => { this.engine.setStatus(next); }`. Bind once; pass the same reference into every recreated context object.
- In `willUpdate`: set `this.dataset.status = this.engine.status`. Reassign `this.avatarCtx = { status: this.engine.status, setStatus: this._setStatus }` only when `!this.hasUpdated` OR `this.engine.status` differs from the currently-stored `this.avatarCtx?.status`.
- In `willUpdate`: if the stored status differs from the previous status (track via a private `_previousStatus` field initialised to `null`), dispatch `new CustomEvent<AvatarStatusChangeDetail>('grund-status-change', { detail: { status }, bubbles: true, composed: false })`. Skip the dispatch when it is the very first update (initial `'idle'` → `'idle'` is a no-op anyway; but for the first loading/loaded/error transition, dispatching is correct — use "previous value differs" as the rule).
- `render()`: `html\`<div part="root"><slot></slot></div>\``.
- Registration guard: `if (!customElements.get('grund-avatar')) { customElements.define('grund-avatar', GrundAvatar); }`.
- Re-exports at bottom: `export { GrundAvatarImage } from './avatar-image'; export { GrundAvatarFallback } from './avatar-fallback'; export type { AvatarStatus, AvatarStatusChangeDetail } from './types';`.

Follow the pattern in `src/components/accordion/accordion.ts` for structure (context provider + engine + stable bound callbacks + willUpdate orchestration).

- [ ] **Step 3: Implement `avatar-image.ts`**

`src/components/avatar/avatar-image.ts` — responsible for rendering an internal `<img>` inside its shadow DOM, forwarding image-related attributes 1:1, reporting status to the root via `ctx.setStatus`, and reflecting its own `data-status`.

Contract:
- `export class GrundAvatarImage extends LitElement` with JSDoc: `@element grund-avatar-image`, `@csspart image - Internal <img>`.
- Static reactive properties for each forwarded attribute (string/null types):
  - `src`, `alt`, `srcset`, `sizes`, `crossorigin`, `referrerpolicy`, `decoding`, `loading`, `fetchpriority` — each declared with `@property()` as `string | null = null`. Non-standard attribute names use `{ attribute: '...' }` only if Lit's default casing doesn't match (use `attribute: 'fetchpriority'` if needed).
- `:host { display: inline; /* inline: defaults hidden; flipped to inline when loaded */ }` static styles. In `willUpdate`, toggle `this.style.display = this.hostStatus === 'loaded' ? ## Deviations

### Task 5 — Event dispatch location
- **Deviated from:** Plan specified dispatching `grund-status-change` inside `willUpdate()` with a `_previousStatus !== null` guard.
- **Actual approach:** Event dispatch moved to `updated()`. `_previousStatus` tracking and null-guard remain but now live in `updated()` instead of `willUpdate()`.
- **Why:** lit-patterns Rule 2 prohibits dispatching `grund-*` events from `willUpdate()` because DOM has not yet been updated when the event fires. Listeners reacting to the event and querying `data-status` would see the old value. `updated()` fires after the DOM reflects the new state.

### Task 6 + 7 — Structural warnings mechanism
- **Deviated from:** Plan specified `queueMicrotask(() => parentElement?.querySelectorAll(...))` inside child `connectedCallback` for duplicate-child warnings.
- **Actual approach:** Structural warnings moved to a `slotchange` listener attached in the root element's `firstUpdated()`. The root element checks `slot.assignedElements()` for duplicate `grund-avatar-image` and `grund-avatar-fallback` children.
- **Why:** lit-patterns Rules 40/42 forbid using `querySelectorAll` for component discovery and require structural composition warnings to be driven by `slotchange` on the container slot. The `queueMicrotask` pattern misses insertions that don't trigger reconnect and couples children to knowledge of their siblings.

### Task 7 — Fallback visibility condition
- **Deviated from:** Plan specified `shouldBeVisible = (this._ctx?.status ?? 'idle') !== 'loaded'` — making fallback visible during `idle`, `loading`, and `error` states.
- **Actual approach:** `shouldBeVisible = (status === 'idle' || status === 'error') && this._delayPassed` — fallback is hidden during `loading`.
- **Why:** Accessibility review found that showing fallback initials alongside an in-progress image load could cause redundant information for screen readers (WCAG 1.3.1). Restricting visibility to `idle` and `error` avoids the overlap.
